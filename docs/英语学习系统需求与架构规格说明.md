# 英语学习系统需求与架构规格说明

更新时间：2026-06-15

## 1. 项目定位

English Learning System 是一个面向个人使用的英语单词复习系统。第一版以离线优先为核心：Android 端安装后可独立学习，电脑端提供 Web 管理与学习界面，电脑也可以在局域网内充当服务端和数据库节点。系统运行时不依赖公网云服务，但从架构边界上预留未来拆分为云端 API、云数据库和对象存储的能力。

第一版默认用户模型为单人多设备：同一个学习者在 Android 手机和电脑 Web 上维护同一份词库、复习进度和学习统计。

## 2. 需求边界

### 2.1 第一版必须支持

- Android 原生安装包：可安装 APK，离线可用。
- 电脑 Web 端：在电脑浏览器访问本地服务，管理词库、学习记录和资源。
- 本地数据库：Android 与电脑端都使用 SQLite 保存核心数据。
- 单词复习闭环：建词、导入、查找、学习、复习、记录结果、生成统计。
- CSV 导入导出：方便手工整理词库和备份迁移。
- 离线释义与音频：通过资源包导入，运行时离线查询与播放。
- 局域网自动发现与同步：同一局域网内发现电脑服务端，配对后自动同步。
- 未来云化边界：同步、存储、资源文件访问必须通过接口抽象，不能和本地实现强耦合。

### 2.2 第一版明确不做

- 公网云账号、云数据库、公网 API 服务。
- 多人账号、权限系统、班级/机构场景。
- 课程体系、语法课、听力课、AI 对话、语音评分。
- Android 后台持续同步。
- Google Play 发布流程。
- 未确认许可证的词典或音频资源内置分发。

### 2.3 文档命名硬性规范

- 项目内所有文档文件名必须使用中文名称。
- Markdown 文档标题必须使用中文标题。
- 英文技术名词可在正文中保留，例如 TypeScript、Expo、Fastify、SQLite。
- 代码文件、包名、API 路径、配置文件名不受此规范约束，继续遵循对应生态约定。

## 3. 技术架构

### 3.1 推荐仓库结构

```text
English Learning System/
  apps/
    mobile/        # Expo React Native Android app
    web/           # React + Vite web client
    server/        # Fastify LAN/local server
  packages/
    shared/        # shared types, schema, SRS, sync contracts
    storage/       # repository interfaces and SQLite adapters
    resources/     # resource pack import/index helpers
  docs/
    英语学习系统需求与架构规格说明.md
```

### 3.2 技术选型

- 语言：TypeScript。
- Mobile：Expo React Native，用于生成 Android 原生安装包。
- Web：React + Vite，用于电脑端管理和学习界面。
- Server：Fastify，作为电脑本地服务端、局域网同步 API 和资源文件服务。
- Database：SQLite。Android 使用 `expo-sqlite`，电脑端使用 SQLite 文件库。
- ORM/Schema：Drizzle ORM，尽量共享 schema、迁移定义和类型。
- Build/Workspace：建议使用 pnpm workspace；项目扩大后再引入 Turborepo。

### 3.3 核心抽象

```text
LearningRepository
  管理单词、词义、词本、卡片、复习记录和统计查询。

ReviewScheduler
  计算每日复习队列、复习后下次复习时间和熟悉度变化。

DictionaryProvider
  从本地资源包读取释义、音标、例句和来源信息。

AudioAssetStore
  管理本地音频文件、哈希、路径、缺失状态和播放 URI。

SyncTransport
  抽象同步传输；第一版实现 LAN HTTP，未来可替换为 cloud HTTP。

DiscoveryProvider
  抽象服务发现；第一版 Android 使用 NSD，电脑服务端使用 mDNS/Bonjour 广播。
```

## 4. 功能需求

### 4.1 Android MVP

- 首次启动创建本地 SQLite 数据库和默认词本。
- 支持手动新增、编辑、删除、搜索单词。
- 支持 CSV 导入导出。
- 支持按词本、标签、熟悉度、到期状态筛选。
- 支持查看单词详情：单词、音标、中文释义、英文释义、例句、标签、音频状态。
- 支持播放离线音频；音频缺失时显示缺失状态，不阻塞学习。
- 支持每日复习队列。
- 复习结果至少包含 `again`、`hard`、`good`、`easy`。
- 支持学习统计：今日复习数、正确率、到期卡片数、连续学习天数。
- 配对电脑服务端后，在 App 启动、回到前台、网络变化和定时前台轮询时自动同步。

### 4.2 Web MVP

- 电脑端启动本地 server 后，浏览器访问 Web。
- 支持词库管理：新增、编辑、删除、搜索、批量导入、导出备份。
- 支持资源包管理：查看已导入资源包、许可证、来源、版本、词条数量、音频数量。
- 支持学习统计面板。
- 支持设备管理：查看已配对设备、最后同步时间、同步状态。
- 支持同步诊断：健康检查、最近同步日志、冲突记录、手动触发同步。

### 4.3 CSV 导入导出

第一版 CSV 至少支持以下字段：

```csv
word,phonetic,definition_zh,definition_en,example,tags,audio_file
```

导入规则：

- `word` 必填。
- `tags` 用英文逗号或分号分隔时必须在导入前规范化。
- 重复单词默认合并到现有词条，保留导入日志。
- `audio_file` 指向资源包内相对路径或本地文件名。
- 音频文件不存在时创建缺失记录，不终止整个导入。
- 每次导入生成导入报告：新增数、更新数、跳过数、错误行、缺失音频。

导出规则：

- 支持导出全部词库或指定词本。
- 导出内容包含学习字段和资源字段，但不直接打包音频二进制。
- 未来可增加完整备份格式，用 zip 打包 SQLite dump、CSV 和音频资源。

## 5. SRS 复习算法

第一版采用简化 SM-2 风格算法，保证结果可解释且易调参。

核心字段：

- `easeFactor`：默认 2.5。
- `intervalDays`：距离下次复习的天数。
- `dueAt`：下次复习时间。
- `lapseCount`：遗忘次数。
- `reviewCount`：复习次数。

复习结果规则：

- `again`：当天或次日重新出现，降低熟悉度，增加遗忘次数。
- `hard`：小幅增加间隔或保持短间隔，略降熟悉度。
- `good`：按正常间隔推进。
- `easy`：更快拉长间隔，提高熟悉度。

复习记录必须 append-only，不能因为卡片状态变化覆盖历史记录。

## 6. 数据模型草案

### 6.1 核心表

- `words`：单词主表，保存拼写、标准化拼写、音标、标签、来源、同步元数据。
- `word_senses`：词义、中文释义、英文释义、例句、词性、来源。
- `audio_assets`：音频哈希、文件路径、格式、来源、许可证、缺失状态。
- `decks`：词本。
- `cards`：复习卡片，连接单词和词本，保存 SRS 状态。
- `review_logs`：复习历史，append-only。
- `sync_devices`：配对设备、设备名称、公钥或 token 摘要、最后同步时间。
- `sync_operations`：本地变更操作日志，用于同步。
- `resource_packs`：词典/音频资源包元数据。
- `settings`：本地设置。

### 6.2 同步元数据字段

需要参与同步的实体至少包含：

```text
id
deviceId
createdAt
updatedAt
deletedAt
version
lastSyncedAt
```

冲突策略：

- `review_logs` 使用 append-only 合并。
- 单词资料、词义和标签第一版使用 last-write-wins。
- 发生冲突时写入 `sync_operations` 或单独冲突日志，Web 端可查看。
- 删除使用 soft delete，即写 `deletedAt`，不能直接物理删除参与同步的数据。

## 7. 局域网发现与同步

### 7.1 服务发现

电脑服务端启动后广播：

```text
serviceType: _elsync._tcp.local
serviceName: English Learning System
port: server port
txt:
  protocolVersion=1
  deviceName=<computer-name>
  pairingRequired=true
```

Android 端通过 NSD 发现服务。发现失败时允许手动输入电脑 IP 和端口作为兜底。

### 7.2 配对流程

- 电脑 Web 端显示一次性配对码或二维码。
- Android 端扫描二维码或输入配对码。
- Android 调用 `POST /pair` 建立信任关系。
- 配对成功后保存设备记录和同步 token。
- 后续同步请求必须携带 token。

### 7.3 自动同步边界

第一版自动同步触发时机：

- Android App 启动。
- Android App 回到前台。
- 设备网络状态变化。
- App 前台运行时按固定间隔轮询。
- Web 端用户手动点击同步诊断。

第一版不做 App 关闭后的后台持续同步。

### 7.4 LAN API

```http
GET /health
GET /.well-known/elsync
POST /pair
GET /sync/changes?since=<cursor>
POST /sync/apply
GET /assets/:sha256
POST /imports/csv
```

API 版本通过 `/.well-known/elsync` 和请求头共同表达。未来云端 API 应尽量复用同一套协议。

## 8. 离线资源方案

### 8.1 资源来源原则

系统可以帮助整理网上开放资源，但文档和数据表必须记录来源、许可证、下载时间、处理脚本版本和字段映射。即使个人使用，也不把许可证不清晰的资源默认内置进安装包。

候选来源：

- WordNet：英文释义和词义网络。
- Wiktionary dump：多语言释义、词性、音标和部分例句。
- Lingua Libre / Wikimedia Commons：发音音频。
- Tatoeba：例句和翻译。

### 8.2 资源包格式

建议资源包使用目录或 zip：

```text
resource-pack/
  manifest.json
  dictionary.csv
  examples.csv
  audio/
    <sha256-or-word>.mp3
```

`manifest.json` 至少包含：

```json
{
  "name": "example-pack",
  "version": "2026-06-15",
  "sources": [],
  "licenses": [],
  "createdAt": "2026-06-15T00:00:00Z"
}
```

## 9. 未来云化边界

第一版不能把业务逻辑写死在局域网 HTTP 或 SQLite 实现里。后续云化路径：

- `SyncTransport`：LAN HTTP 替换为 Cloud HTTP。
- `StorageAdapter`：Server SQLite 替换为 Postgres。
- `AudioAssetStore`：本地文件替换为对象存储。
- `DiscoveryProvider`：NSD 发现替换为账号登录后的设备列表。
- `Pairing`：局域网配对码替换为账号授权或设备授权。

云化后仍应保留 Android 离线数据库；云端只作为同步和备份节点。

## 10. 验收标准

### 10.1 Android 验收

- 断网状态下可新增单词、完成复习、查看统计。
- 重启 App 后数据仍存在。
- 离线音频可播放；缺失音频不崩溃。
- 可导入包含错误行和缺失音频的 CSV，并生成报告。

### 10.2 Web 验收

- 可在电脑浏览器管理词库和资源包。
- 可导入/导出 CSV。
- 可查看同步设备、同步日志、冲突记录。
- 本地服务重启后数据仍存在。

### 10.3 局域网同步验收

- 电脑服务端启动后 Android 可自动发现。
- 配对后 Android 和 Web 可互相同步新增单词。
- 手机复习记录同步到电脑后统计更新。
- 同一单词两端同时修改时，last-write-wins 生效，并留下冲突日志。

### 10.4 验证命令与检查

实现阶段至少需要：

- TypeScript typecheck。
- Shared SRS 单元测试。
- Server API 测试。
- CSV 导入测试。
- Web 端基本交互测试。
- Android 真机或模拟器安装冒烟测试。

## 11. 分阶段实施路线

### Phase 0：工程骨架与共享模型

- 初始化 TypeScript workspace。
- 建立 `shared` 类型、SRS 算法和数据 schema。
- 写 SRS 单元测试和基础 fixture。

### Phase 1：Android 离线单机闭环

- Expo Android app。
- SQLite 本地存储。
- 单词 CRUD、CSV 导入导出、复习队列和统计。
- 本地音频播放。

### Phase 2：电脑 Web 与本地 Server

- Fastify server + SQLite。
- React/Vite Web。
- Web 管理词库、资源包和学习统计。

### Phase 3：局域网发现与同步

- mDNS/Bonjour 广播。
- Android NSD 发现。
- 配对、同步 token、changes/apply API。
- 冲突日志和同步诊断。

### Phase 4：资源包工具链

- 下载/整理开放词典和音频资源。
- 生成 resource pack。
- 导入、索引、缺失资源报告。

### Phase 5：云化预留验证

- 把本地 LAN transport 抽象替换为 mock cloud transport。
- 验证 repository、sync 和 asset store 不依赖具体实现。

## 12. 主要风险与处理

- Android 零配置发现存在设备、系统权限和路由器差异：必须提供手动 IP 兜底。
- 离线音频体积可能快速变大：第一版采用资源包导入，不把大量音频强制打进 APK。
- 许可证不清会影响未来分发：所有资源必须记录来源和许可证。
- 同步冲突复杂度高：第一版只做单人多设备，复习记录 append-only，资料冲突 last-write-wins。
- Android 后台限制较多：第一版只做前台自动同步。
